# miraclefile — 아키텍처 설계

탈중앙 · 프라이버시 우선 파일 업로드/공유 서비스.
릴레이도 저장 노드도 파일 내용을 볼 수 없는 구조.

작성: 2026-07-10
상태: 설계 초안. §5(업로드)·§6(gift wrap 공유·수신)·§7(복호)은 구현됨(`js/`), 암호 리뷰 미수행.
      §4 키 볼트(Argon2id·duress)와 §2 Sia 백엔드·본인 운영 릴레이는 미구현.
위협 모델: 강압(coercion)·감시(surveillance)·관할권 압류를 상정한 하이-쓰렛 사용자

---

## 0. 설계 원칙

S3에서 가져온 세 가지:

1. **메타데이터 평면과 데이터 평면 분리.** 인덱스는 빠른 계층에, 바이트는 느린 계층에.
2. **복제 대신 소거 부호(erasure coding).** 같은 durability에 저장 비용이 훨씬 낮다.
3. **커밋은 인덱스 쓰기가 결정한다.** blob이 올라가도 인덱스 커밋 전에는 파일이 "존재"하지 않는다.

이 프로젝트의 위협 모델에서 추가되는 네 가지:

4. **평문은 클라이언트에만 존재한다.** 서버·릴레이·저장 노드 중 어느 것도 키를 갖지 않는다.
5. **영구 흔적을 원천 배제한다.** 온체인 앵커링 금지. 지울 수 없는 기록을 남기는 백엔드는 채택하지 않는다.
6. **네트워크 신원 보호는 사용자 선택이다.** 전송 익명화(Tor 등)는 권장하되 강제하지 않는다. 기본 전송은 HTTPS이고, IP 노출은 인정된 잔여 위험이다.
7. **강압에 대비한다.** 기기 압류 시 "빈 상태"를 보여줄 수 있어야 한다. 부인가능성이 기능이다.

## 1. 계층 구조

```text
[클라이언트]  ← 유일한 평문 지점. 모든 암복호가 여기서 일어난다.
     │            (아웃바운드는 기본 HTTPS. Tor는 사용자가 켤 수 있는 옵션.)
     │
     ├── 데이터 평면 ──→ Blossom 서버 3~5개 (뒤에 Sia renterd / Storj)
     │                    암호문만 본다. 키가 없다.
     │                    서로 무관한 운영자, 서로 다른 관할권.
     │
     └── 메타 평면 ───→ Nostr 릴레이 (본인 운영 inbox 릴레이 포함)
                          gift wrap만 본다. 내용도 발신자도 모른다.
```

**Nostr은 저장소가 아니다.** 릴레이는 blob을 받지 않으며 이벤트 크기 상한이 보통 수십~수백 KB다.
파일 바이트를 이벤트에 넣는 것은 설계 실패다.
Nostr은 컨트롤 플레인(신원·발견·공유)이고, Blossom 서버가 스토리지 노드다.
S3의 3계층이 그대로 매핑된다.

| S3 계층 | miraclefile |
|---------|-------------|
| 프론트엔드 | 클라이언트 (암복호, 유일한 평문 지점) |
| 인덱스 / 메타데이터 | Nostr 릴레이 |
| 스토리지 노드 | Blossom 서버 (뒤에 Sia/Storj) |

### 관련 규격

| NIP / BUD | 역할 |
|-----------|------|
| NIP-44 v2 | ChaCha20 + HMAC-SHA256 페이로드 암호화. 상한 65,535바이트 → **디스크립터 전용** |
| NIP-59 | Gift wrap. 발신자 신원·타임스탬프 은닉 |
| NIP-17 | 위 둘로 만든 비공개 DM |
| NIP-65 | inbox 릴레이 목록. **본인 운영 릴레이로 관찰 범위 최소화** |
| NIP-98 | HTTP 요청을 nostr 키로 서명 (Blossom 업로드 인증) |
| Blossom (BUD-01~04) | sha256 주소 기반 blob 서버. 업로드/미러/삭제 |

## 2. 스토리지 백엔드 결정: Blossom + Sia (온체인 앵커링 배제)

### 왜 Walrus를 쓰지 않는가

Walrus는 읽기 경로가 공개적이고 에포크 TTL이 내장이라 매력적이다.
그러나 **모든 blob이 Sui에 영구 등록된다** — `blob_id`, `size`, `encoding_type`, 지불 주소, 시각이
지울 수 없게 온체인에 남는다.

이 위협 모델(강압·무흔적)에서 이것은 **단일 결격 사유**다:

- 내용은 암호화돼 있어도 파일의 존재·크기·시각·지불자 주소가 영구 공개된다.
- 삭제해도 온체인 기록은 남는다. 원칙 5(영구 흔적 배제)와 정면충돌.
- 지불 주소가 업로드를 특정 지갑에 묶는다. 자금 흐름 추적으로 신원 연결 위험.

성능·노드 수로는 Walrus가 낫지만, **지울 수 없는 기록을 남기는 순간 이 서비스의 존재 이유가 사라진다.** 배제.

### 왜 Blossom + Sia인가

**데이터 평면 = Blossom 서버 3~5개.**

Blossom(BUD-01~04)은 sha256 콘텐츠 주소 기반 HTTP blob 서버 규격이다.
blob 주소 = 암호문의 sha256. 온체인 앵커링이 전혀 없다.

- **서로 무관한 운영자, 서로 다른 관할권.** 한 곳이 압류돼도 다른 미러가 산다. 단일 법적 타깃이 없다.
- **콘텐츠 주소 지정.** 같은 sha256이면 어느 서버에서 받아도 동일. 서버 하나가 죽어도 미러에서 복원.
- **블록체인 없음.** 지울 수 없는 기록이 생기지 않는다.
- **삭제 가능.** BUD-02 `DELETE`로 최선 노력 삭제.

**Blossom 뒤의 실제 바이트 저장 = Sia `renterd`.**

Blossom 서버는 얇은 어댑터다. 그 뒤에서 `renterd`가 소거 부호화(30/10 리드-솔로몬)와 호스트 분산을 처리한다.
5,699 활성 호스트, 8년+ 가동. blob 내용·메타는 오프체인(계약만 체인).

**읽기 경로 중앙화 문제.**
Walrus 논의에서 Sia의 결격 사유로 "모든 다운로드가 내 renterd 한 대를 통과한다"를 지적했다.
이 위협 모델에서는 그것이 **오히려 통제점**이 된다 — 공개 읽기 경로가 자산이 아니라 부채다.
읽기를 통제하면 다운로드 횟수 제한·즉시 차단을 강제할 수 있다.
그 통과 지점을 원하면 Tor 히든 서비스로도 노출할 수 있다(§3, 선택). 단일 장애점 리스크는 Blossom 미러 다중화로 완화.

### 공개 Blossom 서버의 현실 (실측 2026-07-10)

BUD-01은 blob 타입을 제한하지 않지만, **운영 중인 공개 서버 대부분은 미디어 CDN**이라 raw 암호문을 415로 거부한다.
서명된 업로드로 8곳을 `application/octet-stream`으로 찔러본 결과:

| 서버 | raw octet-stream | 비고 |
|------|:--:|------|
| `nostr.download` | 201 | raw 수용 (단 PNG 스텁 + X-SHA-256 조합엔 409) |
| `blossom.band` | 415 | 바이트 스니핑 |
| `blossom.nostr.build` | 415 | 위와 동일 |
| `24242.io` | 400 | — |
| `cdn.nostrcheck.me` | 400 | 타입 검사 |
| `blossom.f7z.io` | 401 | 화이트리스트로 추정 |
| `cdn.satellite.earth` | 502 | — |
| `blossom.primal.net` | timeout | — |

**해결: PNG 스텁 위장.** 암호문을 유효한 1x1 PNG의 IEND 뒤에 붙이고 `Content-Type: image/png`로 보낸다
(`js/crypto.js`의 `PNG_STUB`). 스니퍼는 진짜 PNG 시그니처를 보고 통과시키고, 이미지 디코더는 트레일링
바이트를 무시하며, 콘텐츠 주소(sha256(전체 바이트))라 바이트를 그대로 돌려주는 서버는 왕복이 정상이다.
libre(`web/src/nostr/mediacrypt.js`)가 검증한 기법을 임의 파일로 확장했다.

이 방식으로 `blossom.band` + `blossom.nostr.build` 둘 다 왕복 확인(이미지 아닌 PDF 페이로드 포함,
바이트 verbatim 저장). **"서로 무관한 운영자" 요구를 공개 서버 2곳으로 충족한다.** 서버가 "이미지"를
트랜스코딩하면 트레일링이 깨지지만, 다운로드 시 sha256 검증(`js/blossom.js` `download`)이 잡아내고
다음 미러로 폴백한다. 관할권을 더 늘리려면 자체 Blossom 인스턴스를 서버 목록에 추가한다.

### 백엔드 추상화

`StorageAdapter` 인터페이스 뒤에 백엔드를 숨긴다. Blossom+Sia가 기본.
위협 모델이 완화되면 Storj·Walrus로 교체 가능하게 코드로 보존.

## 3. 전송: HTTPS 기본, Tor는 선택

**기본 전송은 HTTPS다. Tor는 강제하지 않는다.**

시스템은 Tor 없이 정상 동작한다. 익명 전송이 필요한 사용자는 스스로 켠다.

- 클라이언트 → Blossom 서버: HTTPS. 운영자가 원하면 `.onion` 미러를 추가로 노출할 수 있다.
- 클라이언트 → Nostr 릴레이: HTTPS/WSS. `.onion` 릴레이도 `servers[]`에 섞어 쓸 수 있다.
- 본인 운영 inbox 릴레이: 일반 도메인 또는 `.onion`. 선택.

**IP 노출은 인정된 잔여 위험이다.** 앞의 모든 암호화는 콘텐츠를 숨기지만 IP는 여전히 메타데이터다.
Blossom 서버·릴레이·망 감시자는 "누가 언제 무엇에 접근했나"를 IP로 부분 재구성할 수 있다.
이를 감수하지 못하는 위협 모델이라면 사용자가 Tor Browser·시스템 VPN·`torsocks` 등으로 직접 전송을 감싼다.

구현: 앱은 전송 계층에 대해 불가지론적(transport-agnostic)이다.
Tor 회로 유무를 검사하지 않으며 fail-closed 게이트도 두지 않는다.
`.onion` 호스트를 `servers[]`에 넣으면 Tor 사용 클라이언트에서 그대로 동작한다.

## 4. 키 계층

신원 키와 콘텐츠 키를 **절대 섞지 않는다**.
분리 이유: nsec이 유출돼도 과거 파일이 안 열리고, 파일 키를 넘겨도 신원이 안 넘어간다.

```text
nsec (신원)          — 서명 전용. 암호화에 직접 쓰지 않는다.
  │
  └─ NIP-44 대화키 = HKDF(ECDH(내priv, 상대pub))   ← 디스크립터 봉인용

CK (파일별 콘텐츠 키) — 32바이트 CSPRNG. 신원과 무관. 파일마다 새로 생성.
  ├─ encKey  = HKDF(CK, info="miraclefile/v1/content")
  └─ metaKey = HKDF(CK, info="miraclefile/v1/meta")     ← 파일명·MIME 암호화
```

CK를 AEAD에 직접 넣지 않고 HKDF로 용도 분리한다.
파일명까지 암호화해야 저장 계층이 `report_final_v3.pdf` 같은 것을 못 본다.

### 키 저장 — 강압 대비

원칙 4·7의 핵심. **어떤 키도 기기에 평문 저장하지 않는다.**

- **nsec / 임시 키 평문 저장 절대 금지.**
- **패스프레이즈 파생 키만 세션 중 메모리에 존재.** 종료 시 즉시 파기(zeroize).
  - 마스터 시드는 Argon2id(패스프레이즈, 솔트) → 여기서 nsec·CK 마스터를 파생.
  - 디스크에는 암호화된 볼트만. 패스프레이즈 없이는 복호 불가.
- **부인가능성(deniable) 구조.** 기기 압류·강압 시 "빈 상태"를 제시할 수 있어야 한다.
  - VeraCrypt 히든 볼륨 개념 차용: 하나의 암호화 볼트에 두 개의 패스프레이즈.
    - 정상 패스프레이즈 → 실제 볼트.
    - duress 패스프레이즈 → 미끼(빈/위장) 볼트. 실제 데이터의 존재를 암호학적으로 부인.
  - duress 볼트는 반드시 그럴듯한 내용을 담아야 한다(완전히 비면 오히려 의심 유발).

주의: 부인가능 볼륨은 구현이 극도로 까다롭다. 접근 패턴·타임스탬프·백업 사본으로 히든 볼륨 존재가 새는 사례가 많다.
자작 금지. 검증된 프리미티브(예: 성숙한 히든 볼륨 라이브러리) 위에서만 구현하고 독립 리뷰를 받는다.

## 5. 업로드 흐름

```text
1. CK ← csprng(32)
2. 파일 → 1 MiB 청크로 분할
3. 청크별 AEAD:
     XChaCha20-Poly1305(encKey, nonce_i, chunk_i)
     nonce_i = 20B 랜덤 프리픽스 ‖ 4B 카운터
     마지막 청크는 AAD에 final=1 플래그
4. 암호문 앞에 1x1 PNG 스텁 prepend → blobHash = sha256(스텁‖암호문)  ← 미디어 CDN 통과용
5. 크기 패딩: 다음 크기 사다리 단계까지 (1/2/4/8/16 MiB …)
6. Blossom 서버 3~5곳에 PUT /upload
     인증 = NIP-98 이벤트, 단 일회용 임시 키로 서명
     업로드 완료 후 임시 키는 세션 종료 즉시 파기
7. 디스크립터 조립
8. Nostr 이벤트 발행  ← 커밋 지점
```

### 청크 프레이밍은 직접 발명하지 않는다

`age`의 STREAM 구성(ChaCha20-Poly1305 + 카운터 논스 + 최종 청크 플래그 바이트)을 그대로 따른다.
자작 프레이밍은 절단(truncation)·재정렬 공격에 거의 항상 뚫린다.

### 임시 키로 업로드 서명

Blossom 인증은 nostr pubkey를 노출한다. 본인 nsec으로 서명하면 blob 서버가 "이 사람이 올렸다"를 안다.
파일당 일회용 임시 키를 쓰면 링크 불가. **그 임시 키는 세션 종료 즉시 파기한다.**

이것은 §9의 삭제 트레이드오프와 직결된다: 임시 키를 버리면 나중에 BUD-02 `DELETE`를 못 한다.
이 위협 모델의 선택은 **삭제 능력보다 무연결성(unlinkability)을 우선**한다 —
필요 시 삭제 못 해도 감수. (TTL로 서버 측 자동 만료를 별도 확보.)

### 크기 패딩

8,431,203바이트짜리 암호문은 그 자체로 지문이다. 패딩 없으면 blob 서버가 크기만으로 파일을 식별한다.

### 커밋 지점

blob이 올라가도 Nostr 이벤트 발행 전에는 파일이 존재하지 않는다. S3에서 인덱스 커밋이 객체를 존재하게 만드는 것과 같다.

### 디스크립터 — 진짜 "열쇠"

```json
{
  "v": 1,
  "hash": "<sha256 of ciphertext>",
  "ck": "<base64url, 32 bytes>",
  "servers": ["https://blossom-a.example", "https://blossom-b.example", "https://blossom-c.example"],
  "size": 8388608,
  "realSize": 8431203,
  "meta": "<metaKey로 암호화된 {name, mime}>"
}
```

## 6. 공유 흐름 — Nostr gift wrap 경유만

**링크 모드는 반드시 Nostr gift wrap으로만 전달한다.**
이메일·SMS 등 평문 채널로 링크 자체를 보내는 것을 금지한다 —
링크 = 열쇠이므로, 열쇠를 감시 가능한 채널로 보내는 순간 전체가 무너진다.

### 링크의 형태 (전달은 gift wrap으로)

```text
https://app.example/f/<blobHash>#<base64url(CK)>
                              ↑
                    프래그먼트. 절대 서버로 전송 안 됨.
```

브라우저는 `#` 뒤를 HTTP 요청에 넣지 않는다. 게이트웨이·blob 서버·릴레이 전부 CK를 못 본다.
Firefox Send가 쓰던 방식이고 검증됐다.
단, 이 링크 문자열을 **평문 채널로 보내지 않는다.** 아래 gift wrap 안에 담아 전달한다.

### 전달 = Nostr 모드 (수신자 지정)

```text
rumor    = {kind: 1063, content: JSON(디스크립터 또는 링크), tags:[...]}   ← 서명 안 함
seal     = kind 13,   NIP-44(내priv, 상대pub, rumor),  내 키로 서명
giftwrap = kind 1059, NIP-44(임시priv, 상대pub, seal), 임시 키로 서명
                      timestamp 무작위화 (±2일)
발행 대상 = 수신자의 NIP-65 inbox 릴레이 (가능하면 본인 운영 릴레이)
```

릴레이가 보는 것: `kind:1059`, 난수 pubkey, 무작위 시각, `p` 태그. **내용도 발신자도 못 본다.**

`rumor`를 서명하지 않는 것이 핵심이다. 수신자가 그 이벤트를 유출해도 발신자를 암호학적으로 증명 못 한다 —
부인가능성(deniability) 확보. 강압 상황 대비의 핵심.

### 접근 제한

- **TTL 최소.** 짧게 잡는다.
- **다운로드 1회 제한 엄격 적용.** 첫 성공 다운로드 후 Blossom/게이트웨이에서 즉시 무효화.
- inbox 릴레이는 **본인이 직접 운영**하여 제3자 릴레이의 관찰 범위를 최소화(NIP-65). 원하면 `.onion`로도 노출.

## 7. 조회 · 복호 흐름

```text
1. gift wrap 해제 → 디스크립터/링크 획득 → 프래그먼트에서 CK 추출
2. servers[] 중 하나에서 GET /<blobHash>
3. 스트리밍하며 sha256 누적 검증 (조기 실패용)
4. HKDF(CK) → encKey
5. 청크별 AEAD 복호. 인증 태그 실패 = 즉시 중단
6. 마지막 청크 final 플래그 확인 → 절단 공격 방지
7. 패딩 제거 (realSize까지 절단)
8. Service Worker + ReadableStream으로 디스크에 직접 스트림
9. 다운로드 성공 통지 → 서버 측 1회 제한 소진 처리
```

**6단계를 빠뜨리면 안 된다.** 공격자가 뒤쪽 청크를 잘라내도 각 청크는 개별적으로 유효하므로 클라이언트가 눈치채지 못한다.
`final` 플래그 검사가 유일한 방어선이다.

**8단계는 실무 필수.** 4 GB 파일을 메모리에 버퍼링하면 탭이 죽는다.

## 8. 위협 모델 — 누가 무엇을 보는가

| 주체 | 파일 내용 | 파일명 | 크기 | 발신자 | 수신자 | IP |
|------|:---------:|:------:|:----:|:------:|:------:|:--:|
| Blossom 서버 | ❌ | ❌ | 패딩된 값만 | ❌ (임시 키) | ❌ | ✅ (Tor 쓰면 ❌) |
| Nostr 릴레이 | ❌ | ❌ | ❌ | ❌ (gift wrap) | ✅ `p` 태그 | ✅ (Tor 쓰면 ❌) |
| 링크 수신자 | ✅ | ✅ | ✅ | 링크 출처만 | — | — |
| 망 감시자 | ❌ | ❌ | ❌ | — | — | ✅ (Tor 쓰면 ❌) |
| 기기 압류자 | ❌ (duress 볼트) | ❌ | ❌ | ❌ | ❌ | — |

duress 볼트가 압류자 행을 닫는다. **IP 열은 기본값으로 열려 있다** — 닫으려면 사용자가 전송을 Tor/VPN으로 감싸야 한다(§3).

## 9. 남는 누출 — 정직하게

**1. `p` 태그가 수신자를 노출한다.**
Gift wrap은 발신자를 숨기지 A→B 중 B는 못 숨긴다. 릴레이는 "누군가 이 pubkey에게 뭔가 보냄"을 본다.
표준 해법 없음. 완화: 본인 운영 inbox 릴레이(NIP-65)로 관찰 범위를 좁힌다.
근본 해결은 PIR·믹스넷인데 Nostr 생태계에 없다.

**2. 삭제 능력을 포기했다.**
임시 키를 세션 종료 시 파기하므로 BUD-02 `DELETE`를 나중에 못 한다.
무연결성을 위한 의도적 트레이드오프. 서버 측 TTL 자동 만료로만 회수 가능.
콘텐츠 주소 + N개 미러 특성상 이미 받아간 사본은 통제 불가.

**3. 링크 모드는 전달 가능하다.**
열쇠가 URL에 있으므로 수신자가 무한 재배포 가능. DRM은 불가능. 인정하고 설계한다.
gift wrap 전달 강제·1회 다운로드·최소 TTL로 노출 창을 줄일 뿐.

**4. NIP-44는 대용량용이 아니다.**
65,535바이트 상한. 디스크립터에만 쓰고 파일 본문에는 절대 쓰지 않는다.

**5. IP가 기본으로 노출된다.**
전송 익명화를 강제하지 않으므로 Blossom 서버·릴레이·망 감시자가 클라이언트 IP를 본다.
"누가 언제 무엇에 접근했나"의 부분 재구성이 가능하다. 이 잔여 위험은 설계상 인정된 것이다.
사용자가 Tor를 켜면 닫히지만, Tor도 만능이 아니다 — 글로벌 패시브 관찰자·타이밍 상관·악성 exit·회로 고갈은 남는다.

**6. 부인가능 볼륨은 완벽하지 않다.**
백업 사본·클라우드 동기화·파일시스템 저널·타임스탬프로 히든 볼륨 존재가 샐 수 있다.
운영 수칙(백업 금지, 저널 관리)이 암호만큼 중요하다.

## 10. 스택

```text
저장:       Blossom 스타일 HTTP blob 서버 3~5개
            — 서로 무관한 운영자, 서로 다른 관할권
            — 블록체인 앵커링 없음 (온체인 영구 기록 원천 배제)
            — 뒤에 Sia renterd (소거 부호화·호스트 분산)
            — 임시 키로만 업로드 서명, 임시 키는 세션 종료 즉시 파기
            — 삭제 능력보다 무연결성 우선 (필요시 삭제 못 해도 감수)

전송:       HTTPS 기본. Tor는 사용자 선택 (앱은 transport-agnostic)
            — fail-closed 게이트 없음. `.onion` 호스트도 servers[]에 그대로 사용 가능
            — IP 노출은 인정된 잔여 위험

메타플레인:  Nostr (NIP-44 v2 + NIP-59 gift wrap + NIP-65)
            — inbox 릴레이 본인 직접 운영
            — gift wrap + rumor 미서명 (부인가능성)

키 관리:     신원키/콘텐츠키 분리
            — nsec/임시키 평문 저장 절대 금지
            — 패스프레이즈 파생 키(Argon2id)만 세션 중 메모리, 종료 시 즉시 파기(zeroize)
            — duress 패스프레이즈 → 미끼 볼트 (VeraCrypt 히든 볼륨 개념)

공유:       링크는 반드시 Nostr gift wrap 경유로만 전달
            — 이메일/SMS 등 평문 채널로 링크 전송 금지
            — TTL 최소, 다운로드 1회 제한 엄격

파일 암호화: XChaCha20-Poly1305, age STREAM 프레이밍, 1 MiB 청크
키 유도:     HKDF-SHA256 (콘텐츠), Argon2id (마스터 시드)
blob 주소:   sha256(암호문) — Blossom BUD-01
클라이언트:  WebCrypto + libsodium-wrappers (WASM), Service Worker 스트리밍
백엔드 추상: StorageAdapter 인터페이스 (Blossom+Sia 기본, Storj/Walrus 교체 가능)
```

## 11. 암호 구현 규칙

**직접 짜는 부분은 오케스트레이션뿐이어야 한다.**

AEAD 프레이밍, HKDF, ECDH, gift wrap, Argon2id, 히든 볼륨은 전부 검증된 라이브러리를 호출한다:

- `libsodium-wrappers` — XChaCha20-Poly1305, HKDF, Argon2id
- `nostr-tools` — `nip44`, `nip59`
- Blossom 클라이언트 — BUD-01~04

논스 생성·청크 프레이밍·히든 볼륨을 직접 구현하면 거의 확실히 취약점이 생긴다.

**프로덕션 전 독립 암호 리뷰를 받는다.**
이 설계는 검토된 패턴들의 조합이지만, 조합 자체는 아직 검토된 적이 없다.
특히 부인가능 볼륨은 실측 검증이 필수다.

## 12. 구현 순서

1. 청크 AEAD 인코더/디코더 + 테스트 벡터 (절단·재정렬 공격 테스트 포함)
2. 디스크립터 직렬화 / 역직렬화
3. `StorageAdapter` 인터페이스 + Blossom 구현
4. 업로드 클라이언트 (패딩 + 임시 키 파기)
5. 키 볼트 (Argon2id 파생 + zeroize + duress 볼트)
6. 링크 모드 + Service Worker 스트리밍 다운로드 + 1회 제한
7. Nostr 모드 (gift wrap, 본인 운영 inbox 릴레이)
